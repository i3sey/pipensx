## 1. Гигиена репозитория и сборки — P0

### 1.1. Out-of-tree сборка для Makefile.pc (S)
`.o`-файлы лежат прямо в `src/core/`, `src/app/`, `src/install/`,
`tests/` — рядом с исходниками. Перевести Makefile.pc на `build-pc/`
(зеркальная структура объектов), бинари тестов — в `build-pc/tests/`.
Заодно убрать `tools/__pycache__/`.

### 1.2. Чистый корень репо (S)
В корне: `pipensx.nro` 21.5M, `pipensx` 134K, три лога на ~900K суммарно
(`pipensx.log`, `new_pipensx.log`, `new2pipensx.log`). Всё untracked, но
мешает. Публикацию NRO перенести в `dist/` (поправить `deploy-switch.sh` и
шаг `cp` из инструкции сборки), логи — в `logs/` или удалить после разбора.

### 1.3. Одна система сборки (M)
Сейчас три: CMakeLists.txt (Switch + golden), Makefile.pc, Makefile.switch
(обёртка). Свести PC-сборку и тесты в CMake (цель `pipensx_pc`,
`ctest`), Makefile.pc оставить тонкой обёрткой или удалить. Меньше дрейфа
флагов между окружениями.

### 1.4. Ротация логов на устройстве (S)
`log_init(LogPath)` пишет один файл без предела — на SD он растёт
бесконечно (уже сотни KB за сессию). Ротация: 2 файла по N МБ,
переименование при старте.

## 3. Надёжность и безопасность — P1 (частично P0)

### 3.1. Fuzzing сетевых парсеров (M) — самое важное в разделе
`bencode.c`, `metainfo.c`, peer wire (`peer.c`), ut_metadata-фреймы в
`magnet_resolver.cpp` парсят байты от **недоверенных пиров**. libFuzzer-цели
на каждый парсер + корпус из реальных .torrent/хендшейков, джоб в CI
(короткий прогон) и локальный длинный. Любая находка — это то, чем чужой
пир может уронить или захватить консоль.

### 3.2. Path traversal из .torrent (S)
Аудит: имена файлов из info-dict (`../`, абсолютные пути, `sdmc:`-префиксы)
должны жёстко санитизироваться до записи на SD. Если уже есть — покрыть
тестом с враждебным metainfo; если нет — добавить.

### 3.3. Подпись каталога ed25519 (M) — П3.2 из RF_ACCESS_PLAN
С antizapret-прокси и будущими зеркалами MITM-поверхность растёт, а каталог
диктует, какие info-hash ставить на консоль. Подпись — обязательное условие
для 3.4 и для OTA-обновлений каталога. Плюс П3.1 (зеркала) следом.

### 3.4. OTA-обновление каталога (M)
Сейчас каталог — снапшот в RomFS. Фоновый фетч свежего `catalog.json`
(через antizapret при необходимости), верификация подписи, атомарная
подмена на SD. Бонус: дельта вместо полного файла.

### 3.5. Самообновление приложения (M)
Проверка GitHub Releases (тег > текущей версии) → скачать NRO → записать
рядом → предложить перезапуск. Стандарт для взрослого homebrew; закрывает
цикл «выпустили фикс — пользователь получил».

## 4. Фичи ядра — P2 («чтобы было круто»)

### 4.1. Magnet-ссылки вручную (M)
README: «Magnet links … not supported» — при том, что весь резолвер BEP 9/10
уже есть и live-verified. Осталась только UI-обвязка: ввод через
software keyboard applet (+ вставка из буфера обмена, если доступно) →
`MagnetResolver` → импорт. Дёшево относительно ценности.

### 4.2. Выбор файлов внутри торрента (L)
Вторая строчка того же README. Требует: маску файлов в piece-планировщике
(качать только пересекающие выбранное), UI-чеклист на карточке/пикере.
Ядро (`piece.c`, `torrent.c`) затрагивается всерьёз — отдельный эпик.

### 4.3. Параллельные загрузки (L)
Сейчас FIFO, одна активная. Минимум: 2–3 одновременных с общим лимитом
пиров/RAM (stream_ram_budget уже умеет бюджетировать). Осторожно:
конфликтует с sequential-install-потоком — стриминговая установка
осмысленна только для одной задачи, остальные — download-only.

### 4.4. Сидирование (M)
После завершения — раздавать (опция, лимит ratio/времени). Отдача уже
частично есть на уровне протокола (unchoke/upload path в peer.c —
проверить полноту). Этично для экосистемы трекера, и «настоящий клиент».

### 4.5. Шифрование пиров MSE/PE + uTP (L)
Для РФ-сценария: DPI режет открытый BitTorrent-хендшейк. MSE/PE (RC4)
прячет протокол; uTP (BEP 29) дополнительно уводит трафик в UDP и лучше
живёт за CGNAT. Логичное продолжение RF_ACCESS_PLAN после П1.2 (PEX).

### 4.6. IPv6 (M)
`net.c`/tracker/DHT сейчас v4. У части РФ-провайдеров v6 не фильтруется —
это и фича, и обход. compact6 в трекер-ответах, DHT по BEP 32.

## 9. Тир Fable — сложные и новые эпики

Задачи с протокольной/архитектурной работой сразу на несколько подсистем —
для сильной модели. Эпики из раздела 4 (4.2 выбор файлов, 4.3
параллельность, 4.5 MSE/uTP, 4.6 IPv6) и PERF 3.3 (многоядерный
install-конвейер zstd→AES→SHA→ncm на 3 ядрах) — тоже этого тира; ниже
только то, чего нет ни в одном плане.

### F-A. Фоновая загрузка: sysmodule + Tesla-оверлей (XL)
Killer feature: качать, пока пользователь играет или консоль спит.
Торрент-движок выносится в системный модуль (демон с жёстким бюджетом
~3–8 МБ RAM), приложение становится UI-клиентом через собственный
IPC-сервис, статус — Tesla-оверлей поверх любой игры. Движок благоприятен
для переноса: чистый C, single-threaded poll(), малый футпринт. Главный
риск — curl/mbedtls в sysmodule; трекер-анонсы, возможно, переписать на
минимальный собственный HTTP-клиент. Riskiest-first: прототип «движок в
sysmodule качает один торрент на SD».

### F-B. Транзакционная установка с резюмом (L) ✅
Прерванная установка большого NSZ (сон, краш, разряд) сейчас теряет
прогресс стрим-цикла. Журнал установки: персистентные ncm-placeholder'ы +
чекпоинт позиции потока (счётчики zstd/AES, промежуточное SHA-состояние) →
после рестарта продолжить с чекпоинта, не с нуля. Требует аккуратной
модели состояний ncm и верификации уже записанного хвоста.
Сделано: `install_journal.{hpp,cpp}` (bencode-журнал per-task),
checkpoint/restore в `PackageStream`/`NczDecoder`, suspend/resume в
backend'ах (PC file-based, Switch ncm-placeholder'ы), пропуск
перезакачки consumed-префикса через `sink_ready_bytes` в storage,
резюм-виринг в `PackageCoordinator`/`DownloadManager`; тесты
`test_install_journal` + чекпоинт-кейсы в `test_package_stream`.

### F-C. WebSeed BEP 19 — HTTP-источники как «пиры» (L)
Каталожная запись получает HTTP-зеркало; загрузчик смешивает ranged-HTTP и
обычных пиров в одном piece-планировщике (скорость источника — вход
rate-matched гейта из PERF 7.1). Синергия с РФ-планом: через antizapret
качает даже при нуле живых пиров.

### F-D. Установка NCA-за-NCA вне глобального порядка (L)
Стрим-установка сейчас хочет почти-последовательный порядок кусков всего
пакета. Ранний парс PFS0/CNMT уже есть (PERF 3.4) — расширить до
независимых очередей приоритета по каждому NCA: NCA коммитится, как только
готов его диапазон. Меньше reorder-RAM (OOM-риск в applet), меньше стоек
гейта, быстрее большие апдейты.

### F-E. Полный choker + tit-for-tat при сидировании (M)
База — 4.4; Fable-часть: честный choker по BEP 3 (4 unchoke-слота,
optimistic unchoke с ротацией 30 с, anti-snubbing), super-seeding.
Взаимность пиров ускоряет и загрузку, не только отдачу.

### F-F. BitTorrent v2 / hybrid (BEP 52) (XL)
SHA-256 + merkle-деревья: пофайловая верификация, иммунитет к
SHA-1-коллизиям, hybrid-торренты. Трогает bencode/metainfo/piece/peer
одновременно. Делать после F-C/F-D — ценность растёт по мере появления
v2 на трекерах.

### F-G. Switch↔Switch по локалке (XL, исследовательская)
Отдать другу уже скачанное без интернета: локальный swarm через ldn
(local wireless) или LAN + mDNS-дискавери. Протокол уже есть (BT) —
нужна транспорт-абстракция в `net.c` + UI «поделиться».

Порядок по соотношению ценность/усилия:
**F-B → F-C → F-E → F-D → F-A → F-F → F-G.**

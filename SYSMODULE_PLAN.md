# F-A. Фоновая загрузка: sysmodule + Tesla-оверлей — план реализации

Дата: 2026-07-06. Детализация пункта F-A из `IMPROVEMENT_PLAN.md` (раздел 9).
Все утверждения о коде подтверждены по исходникам; ссылки — `файл:строка`.

Суть: торрент-движок выносится в системный модуль Atmosphere (демон с жёстким
бюджетом RAM), приложение становится IPC-клиентом, статус и управление — через
Tesla-оверлей поверх любой игры.

---

## 0. Анализ текущего кода: что помогает и что мешает

### 0.1 Благоприятно для переноса

- **Движок — чистый C, компактный.** `src/core` = 4 843 строки (`.c`+`.h`),
  зависимостей вне libc/сокетов почти нет.
- **Однопоточный цикл.** Вся работа — `torrent_tick()` (`torrent.h:64`) поверх
  `poll()`. Побочные потоки только два: announce-worker
  (`torrent.c:132–133,838`, блокирующий curl) и UPnP (`torrent.c:149`,
  miniupnpc). Оба опциональны по своей природе.
- **Узкий публичный API.** Всего 9 функций: `torrent_create[_ex]/destroy`
  (`torrent.h:45–52`), `add_initial_peers` (`:57`), `tick` (`:64`), `stat`
  (`:67`), `last_error` (`:68`), `set_strict_lookahead` (`:74`),
  `set_rate_freeze` (`:82`). Поверхность будущего IPC фактически уже описана
  типами `torrent_stat_t` и `torrent_options_t`.
- **UDP-трекеры (BEP15) уже реализованы без curl** — `tracker.c:254–335`.
  Это готовый запасной путь, если curl в sysmodule не приживётся.
- **Верификация из storage уже есть.** `piece_mgr_verify_piece` читает кусок
  с диска (`piece.c:204–218`), `piece_mgr_verify_all` (`piece.c:280`) —
  фундамент для resume и для будущего write-through-режима (фаза 5).
- **Storage — обычный `fopen`** (`storage.c:91–95`): в sysmodule работает
  через `fsdevMountSdmc()` без правок.
- **Готовый headless-шаблон.** `src/main_pc.c` — CLI-цикл
  «metainfo → torrent_create → tick до завершения», ровно то, что нужно
  демону.
- **Персист-паттерны готовы.** bencode-сериализатор (`src/core/bencode.c`),
  атомарная запись tmp+rename (`download_manager.cpp:1418–1444`), формат
  очереди `queue.bencode` (`download_manager.cpp:1162`).

### 0.2 Мешает / требует решения

| Проблема | Где | Последствие для sysmodule |
|---|---|---|
| curl+mbedtls в HTTP-announce | `tracker.c:42–126` | главный риск плана: heap, entropy (csrng), поведение вне applet-контекста |
| antizapret-маршрутизация завязана на CURL-handle | `antizapret.c:302–345` | без curl фоновые анонсы теряют обход блокировок (см. `RF_ACCESS_PLAN.md`) |
| `peer_t` весит ~285 КиБ: inline `rbuf` 256 КиБ (`peer.h:10,56`) + `sbuf` ~16 КиБ (`peer.h:9`) + pipeline (`peer.h:6,72–73`) | `peer.h` | `MAX_PEERS 96` (`peer.h:7`) лениво аллоцируются (`torrent.c:557–558`) → худший случай ~27 МБ; нужен cap |
| Буфер целого куска на каждый недокачанный piece: `calloc(piece_length)` | `piece.c:137,157` | кусок 4 МиБ × N параллельных кусков; нужен строгий порядок с малым lookahead |
| Запрос SO_RCVBUF 256 КиБ на сокет | `net.h:28–29,38–40` | в sysmodule bsd-сервис с малым tmem откажет; путь уже best-effort с fallback 128 КиБ — безопасно, но надо сжать конфиг |
| Stream-install не хранит полной копии (режимы SINK/SKIP, `storage.h:9–12`) | install-конвейер | sysmodule **не может** продолжать stream-install-задачи: конвейер zstd→AES→SHA→ncm в бюджет не влезает |
| Порт 51413 захардкожен в вызове | `download_manager.cpp:1690` | конфликт, если движки приложения и sysmodule живут одновременно |
| Очередь задач — в приложении (`queue.bencode`) | `download_manager.cpp:1162` | нужен один владелец состояния фоновых задач |

### 0.3 Честная оговорка про «качать, пока консоль спит»

В реальном сне (SC7, suspend-to-RAM) **все процессы, включая homebrew
sysmodule, приостановлены**. Официальные фоновые загрузки используют
системный low-power-режим, недоступный homebrew. Поэтому фича формулируется
так:

- **качать, пока пользователь играет в любую игру** — основная ценность;
- **пережить сон без потерь**: по psc-уведомлению о засыпании — сохранить
  состояние и закрыть сокеты, по пробуждению — переанонс и автоматическое
  продолжение. Загрузка во сне — явная не-цель (§7).

## 1. Целевая архитектура

Три бинаря, общий код движка не форкается:

```
pipensx.nro                      — приложение (есть; становится IPC-клиентом
                                   для фоновых задач)
atmosphere/contents/<TID>/       — sysmodule: exefs.nsp + flags/boot2.flag
                                   + toolbox.json
switch/.overlays/pipensx.ovl     — Tesla-оверлей (libtesla + nx-ovlloader)
```

Структура исходников:

```
src/sysmodule/main.c        — init сервисов, heap, главный цикл
src/sysmodule/ipc_server.c  — cmif-сервер сервиса "pipensx"
src/sysmodule/task_queue.c  — очередь задач + персист (тестируема на PC)
src/sysmodule/engine_host.c — обвязка torrent_create/tick, бюджеты
src/ipc/pipensx_ipc.h       — общий протокол (cmd id, POD-структуры) —
                              единственный источник правды для всех трёх
                              бинарей
overlay/                    — исходники .ovl
```

### 1.1 Разделение ответственности

| Компонент | Делает | НЕ делает |
|---|---|---|
| sysmodule | download-to-SD (`STORAGE_FILE_DISK`), очередь, resume, psc sleep/wake, анонсы/DHT | stream-install (ncm), magnet resolve, каталог, UI |
| приложение | всё как сейчас + IPC-клиент: постановка фоновых задач, отображение их статуса; magnet → `.torrent`-файл на SD для sysmodule | владение фоновыми задачами |
| оверлей | список задач, скорость, pause/resume, вкл/выкл демона | постановка новых задач |

Разруливание сети: sysmodule слушает 51413, движок приложения
(stream-install, пока приложение открыто) — 51414. Команды
`ClaimNetwork/ReleaseNetwork` в протоколе зарезервированы на случай, если
двух портов окажется мало (memory contention, лимиты bsd).

Состояние фоновых задач хранит **только sysmodule**
(`sdmc:/switch/pipensx/sysmodule-queue.bencode`, атомарная запись по образцу
`download_manager.cpp:1418–1444`). Приложение отображает их по IPC и в свой
`queue.bencode` не дублирует — ссылки по id задачи.

### 1.2 Сервисы, нужные sysmodule (вместо ncm/ns/es из `main_switch.cpp:147–167`)

`sm`, `fsdev (sdmc)`, `bsd/socket` (сжатый конфиг, см. §2), `sfdnsres`
(резолвер, входит в socket-конфиг), `csrng` (entropy для mbedtls), `psc`
(sleep/wake), `nifm` (наличие сети, ожидание восстановления), `time`.
Heap — `svcSetHeapSize` (см. бюджет). Потоки тика — приоритет ниже
игрового, пин на системное ядро (core 3), чтобы не красть CPU у игры.

## 2. Бюджет памяти

Цель из F-A: 3–8 МБ. Две ступени: прототип ≤ 8 МБ (движок без правок),
оптимизированный ≤ 4 МБ (фаза 5).

### Прототип (движок как есть, только runtime-ручки)

| Статья | Расчёт | Объём |
|---|---|---|
| Пиры | cap 12 × ~285 КиБ (`peer.h:7,10,56`) | ~3.4 МБ |
| Piece-буферы | strict order, lookahead 1 → 1 × piece_length (типично 4 МиБ) (`piece.c:137`, `torrent.h:13–16`) | ~4 МБ |
| DHT + трекеры + очередь + bencode | `dht.c`, `tracker.c` | ~0.3 МБ |
| Стеки потоков (main, tick, announce) | явные размеры, 32–128 КиБ | ~0.3 МБ |
| **Итого heap** | | **~8 МБ** |

Socket-конфиг (`SocketInitConfig`): tcp rx/tx 32/16 КиБ на сокет,
sb_efficiency 2, ~12 TCP + 2 UDP (DHT, UDP-трекер) + резолвер; tmem
~0.5–1 МБ. Запрос SO_RCVBUF 256 КиБ (`net.h:28`) в sysmodule-сборке не
выполнять (compile-time knob) — путь и так best-effort.

### Оптимизация (фаза 5)

| Мера | Эффект |
|---|---|
| `PEER_RECV_BUFFER_SIZE` → 64 КиБ через `#define`-override для sysmodule-сборки | пиры: 3.4 → ~1.2 МБ |
| Write-through: блоки писать в storage по приходу, верифицировать read-back'ом (путь уже есть: `piece.c:204–218`) | piece-буферы: 4 МБ → ~0 (плюс небольшой буфер верификации) |
| **Итого** | **~2–3 МБ** |

Компромисс осознанный: lookahead 1 и 12 пиров режут скорость относительно
foreground-движка (96 пиров, свободный порядок). Для фоновой загрузки это
приемлемо; критерий — ≥50% скорости приложения (§6).

## 3. IPC-протокол (сервис `pipensx`)

Имя сервиса `pipensx` — 7 символов, лимит sm (8) соблюдён. Реализация —
cmif-сервер на чистом libnx (без libstratosphere: не тащить второй тулчейн
в CMake-сборку). Все структуры — POD фиксированного размера в
`src/ipc/pipensx_ipc.h`, версия протокола проверяется первой командой.

| Cmd | Имя | In | Out |
|---|---|---|---|
| 0 | GetVersion | — | u32 protocol, u32 build |
| 1 | ListTasks | — | buffer: TaskInfo[] |
| 2 | AddTask | buffer: путь `.torrent`, outdir | u64 task_id |
| 3 | PauseTask | u64 id | rc |
| 4 | ResumeTask | u64 id | rc |
| 5 | RemoveTask | u64 id, u8 delete_data | rc |
| 6 | GetTaskDetail | u64 id | TaskDetail (телеметрия) |
| 7 | SetLimits | u16 max_peers, u8 rate_freeze | rc |
| 8 | ClaimNetwork | — | rc (пауза всех, закрыть сокеты) |
| 9 | ReleaseNetwork | — | rc |
| 10 | GetLogTail | u32 max_bytes | buffer |

`TaskInfo`: id, infohash (20 байт), имя (усечённое, fixed), статус (enum:
Queued/Downloading/Verifying/Paused/Complete/Error), completed/total (u64),
speed_bps (u32), peers/dht_good (u16) — прямое отражение `torrent_stat_t`
(`torrent.h:23–37`). `TaskDetail` добавляет last_error и счётчики анонсов.

Push-событий нет: приложение и оверлей поллят (1 Гц оверлей, 0.5 Гц
приложение) — дёшево и просто.

## 4. Фазы (riskiest-first)

### Фаза 0 — PoC переносимости, headless NRO (M)

Прототип из F-A («движок в sysmodule качает один торрент на SD»), но ещё
дешевле: собрать headless-цикл по образцу `src/main_pc.c` как **обычный
NRO без borealis** (`pipensx_headless.nro`): берёт `.torrent` с SD, качает
в `sdmc:/switch/pipensx/downloads`, лог в файл. Один рабочий поток, cap
12 пиров, lookahead 1.

Проверяет разом: движок + curl/mbedtls + bsd-сокеты + fopen-storage на
Switch вне GUI-контекста, и даёт первый замер heap (телеметрия
`svcGetInfo(InfoType_UsedHeapMemorySize)` в лог раз в 10 с).

Выход: работает / упёрлись в curl → сразу включаем запасной путь
(UDP-трекеры `tracker.c:254` + мини-HTTP-announce, см. фазу 5) и только
потом идём дальше.

### Фаза 1 — скелет sysmodule + перенос PoC (L)

- Build-инфра: отдельный CMake-таргет; ELF → `main`/`main.npdm`
  (npdmtool из switch-tools, json-дескриптор: набор svc, доступ к fs+bsd)
  → `exefs.nsp` (build_pfs0); title id из homebrew-диапазона, например
  `0x00FF0000504E5358` («PNSX»); `flags/boot2.flag` для автозапуска;
  `toolbox.json` — чтобы пользователь мог выключить модуль стандартными
  средствами.
- Init по §1.2, `svcSetHeapSize(8 МБ)`, лог
  `sdmc:/switch/pipensx/sysmodule.log` (переиспользовать `log_init`).
- cmif-сервер: только `GetVersion`; smoke-клиент в приложении (строчка в
  экране диагностики).
- Внутрь — цикл фазы 0. Критерий: торрент докачивается, пока запущена игра.

### Фаза 2 — очередь, персист, sleep/wake (L)

- `task_queue.c`: несколько задач, последовательное выполнение (одна
  активная — бюджет), статусы, `sysmodule-queue.bencode` через
  `bencode.c` + atomic rename.
- Resume: при старте задачи — `piece_mgr_verify_all`-путь (уже в движке),
  троттлинг тика при верификации, чтобы не грузить CPU под игрой.
- psc: уведомление о сне → пауза, сохранение, закрытие сокетов;
  пробуждение → nifm-ожидание сети → пересоздание движка задачи и
  переанонс.
- Обработка OOM/ошибок SD (место проверять при AddTask — паттерн
  `install_space.cpp`), лимит одновременных partial pieces.
- Тесты: `task_queue` + сериализация протокола собираются и гоняются на PC
  (паттерн `Makefile.pc`, `tests/`).

### Фаза 3 — интеграция приложения (M)

- IPC-клиент (общий C-код для приложения и оверлея).
- `DownloadManager`: новый режим задачи «Background» → `AddTask` вместо
  собственного движка; `magnet_resolver` сохраняет полученный metainfo в
  `.torrent`-файл для передачи.
- UI: фоновые задачи в общем списке загрузок (источник — IPC-поллинг),
  пометка «фон».
- Арбитраж: порт приложения для stream-install → 51414; фоновые задачи не
  трогаются при запуске приложения.
- Поведение при выходе из приложения: download-to-SD-задачи предлагается
  переводить в фон; stream-install-задачи остаются паузой до следующего
  запуска (не-цель §7, journal F-B уже даёт resume).

### Фаза 4 — Tesla-оверлей (L)

- vendor: libtesla; требование для пользователя — nx-ovlloader + Tesla-Menu
  (документировать в README).
- `.ovl`: список задач (имя, прогресс-бар, скорость, пиры), pause/resume,
  «выключить демон до перезагрузки».
- Тот же `pipensx_ipc.h`; поллинг 1 Гц только пока оверлей открыт.

### Фаза 5 — память и отказ от curl (M/L, по результатам фаз 0–2)

- `PEER_RECV_BUFFER_SIZE`/`NET_TCP_RECEIVE_BUFFER_SIZE` — compile-time
  knobs для sysmodule-сборки (64 КиБ / не запрашивать).
- Write-through + read-back-верификация (расширение `piece.c`; путь чтения
  уже есть — `piece.c:204–218`) → отказ от полных piece-буферов.
- Мини-HTTP/1.1 announce-клиент (plain GET; HTTPS-трекеры — mbedtls
  напрямую, без curl) → curl выкидывается из sysmodule; UDP-трекеры уже
  независимы.
- antizapret в sysmodule: без curl нужна собственная маршрутизация
  (SOCKS5 handshake — просто; HTTPS-proxy — сложно, и её и так нет в
  switch-curl, см. `antizapret.c:345`). Согласовать с `RF_ACCESS_PLAN.md`;
  до тех пор фоновые анонсы работают без обхода, DHT это частично
  компенсирует.

### Фаза 6 — полировка и дистрибуция (S/M)

- CI: сборка всех трёх артефактов, релизный zip с раскладкой
  `atmosphere/…`, `switch/…`, `switch/.overlays/…`.
- Телеметрия heap/скорости в лог, крашлог-обвязка для sysmodule
  (по образцу `switch_crashlog.c`).
- README: установка, ограничения (сон), toolbox.

MVP = фазы 0–3 (фон управляется из приложения, без оверлея). Фазу 4 можно
вести параллельно после фазы 1.

## 5. Риски

| Риск | Вер-ть | Митигация |
|---|---|---|
| curl/mbedtls в sysmodule: heap-пики, entropy, скрытые зависимости от applet-окружения | высокая | фаза 0 отсекает раньше всего остального; готовый fallback: UDP-трекеры (`tracker.c:254`) + мини-HTTP (фаза 5) |
| Ожидание «качает во сне» не сбудется (SC7) | факт | честный скоуп (§0.3, §7), psc-пауза/резюм; README |
| Лимиты bsd-сервиса для sysmodule (число сокетов, tmem) | средняя | cap 12 пиров, сжатый SocketInitConfig; SO_RCVBUF-путь уже best-effort (`net.h:38–40`) |
| Давление на SYSTEM-пул памяти Atmosphere (8 МБ — заметно) | средняя | ступень ≤4 МБ (фаза 5); toolbox.json — модуль выключаем; одна активная задача |
| Конкуренция двух движков (порт, память, swarm) | средняя | фиксированные порты 51413/51414; резерв ClaimNetwork/ReleaseNetwork |
| Расщепление состояния очередей app/sysmodule | средняя | единственный владелец — sysmodule; приложение не персистит фоновые задачи |
| Продолжение stream-install в фоне невозможно (SINK/SKIP без копии, `storage.h:9–12`) | факт | не-цель; журнал F-B даёт resume при следующем запуске приложения |
| Стартовая верификация большого торрента съедает CPU под игрой | средняя | троттлинг тика, низкий приоритет потока, пин на core 3 |
| Износ/нехватка SD (фон качает полный образ) | низкая | проверка места при AddTask; статус Error с внятным текстом |

## 6. Критерии приёмки

1. **PoC (фаза 0/1):** торрент ≥1 ГБ докачивается на SD при запущенной
   игре; скорость ≥50% от foreground-загрузки того же торрента в
   приложении; heap peak ≤ 8 МБ по телеметрии.
2. **Sleep/wake (фаза 2):** сон ≥10 мин посреди загрузки → после
   пробуждения загрузка продолжается сама; финальная верификация проходит,
   файлы не повреждены.
3. **Интеграция (фаза 3):** задача, поставленная из приложения, качается
   после выхода из приложения и видна при повторном запуске со статусом.
4. **Оверлей (фаза 4):** открывается поверх игры; pause/resume действуют
   ≤1 с; заметного влияния на FPS игры нет.
5. **CI (фаза 6):** три артефакта собираются в CI; PC-тесты очереди/протокола
   зелёные.

## 7. Явные не-цели

- Stream-install (zstd→AES→SHA→ncm) внутри sysmodule — остаётся в
  приложении.
- Продолжение stream-install-задач в фоне (вариант «фон докачивает в файл
  на SD, приложение потом ставит с SD» — возможное расширение, отдельным
  решением: двойной расход места).
- Magnet resolve / каталог / метаданные в sysmodule.
- Загрузка в реальном сне (SC7).
- Сидирование как отдельная фоновая функция (движок отдаёт куски и так,
  но раздача не управляется).

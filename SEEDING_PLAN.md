# F-E. Отдача и честный choker (tit-for-tat) — план реализации

Дата: 2026-07-06. Детализация F-E из `IMPROVEMENT_PLAN.md` (раздел 9) вместе
с его базой — эпиком 4.4 «Сидирование». Мотивация оценена через призму
взаимодействия с RuTracker. Ссылки на код — `файл:строка`.

---

## 0. Анализ: текущее состояние и почему это важно для RuTracker

### 0.1 Клиент сегодня — чистый лич, и он портит учёт трекера

Подтверждено по коду:

- **Входящие `MSG_REQUEST` не обрабатываются вообще.** В диспетчере
  сообщений (`peer.c:327–397`) есть cases для CHOKE/UNCHOKE/INTERESTED/
  NOT_INTEREST/HAVE/BITFIELD/PIECE/EXTENDED/PORT — ветки `MSG_REQUEST`
  (`peer.h:19`) нет. Отдачи блоков не существует как пути кода.
- **Пиры никогда не анчоукаются.** `peer_choked = 1` устанавливается при
  создании (`peer.c:91`) и никогда не сбрасывается — `MSG_UNCHOKE` мы не
  отправляем ни одному пиру.
- **Announce лжёт про отдачу и события.** Строка запроса:
  `&uploaded=0&downloaded=%lld&left=%lld&compact=1&event=started&numwant=200`
  (`tracker.c:100–102`): `uploaded` захардкожен нулём, `event=started`
  уходит на **каждый** анонс, `completed`/`stopped` не отправляются
  никогда.
- **Extended handshake заявляет `upload_only=0`** (`peer.c:123`) — то есть
  «я раздаю», что неправда.
- Чтение с диска для отдачи готово: `storage_read` уже обслуживает
  верификацию (`piece.c:218,256`).

### 0.2 Что это значит на RuTracker

- **Рейтинг аккаунта.** RuTracker считает ratio по announce-репортам
  клиента. pipensx всегда репортит `uploaded=0` → каждая скачанная игра
  ухудшает рейтинг пользователя; при низком рейтинге трекер ограничивает
  число одновременных закачек.
- **Статистика раздач.** `event=started` на каждом переанонсе и отсутствие
  `stopped`/`completed` дают трекеру фантомные сессии и некорректный учёт
  сидов/личей на раздаче.
- **Здоровье экосистемы.** Рои старых игр на RuTracker и так тонкие;
  клиент, который берёт и не отдаёт, ускоряет смерть раздач — а значит и
  деградацию самого сценария pipensx (см. WEBSEED_PLAN.md — компенсация,
  но не замена).
- **Скорость самой загрузки.** Tit-for-tat — механика взаимности BEP 3:
  пиры отдают охотнее тем, кто отдаёт им. Лич без отдачи живёт только на
  optimistic-слотах чужих клиентов — на тонких роях это заметно медленнее.

Вывод: **фаза 0 (протокольная честность) и фаза 1 (отдача) — самые
дешёвые и самые важные для RuTracker правки во всём Fable-тире.**

### 0.3 Ключевое ограничение: stream-install несидируем

Стрим-установка не хранит полной копии данных (режимы SINK/SKIP,
`storage.h:9–12`) — после установки игры раздавать нечего. Следствия:

- Отдача во время загрузки (tit-for-tat) работает для **всех** режимов —
  пока кусок ещё в piece-буфере/на диске.
  Оговорка: в stream-режиме отдать можно только куски, ещё не
  потреблённые конвейером; окно маленькое → вклад скромный.
- Полноценное сидирование после завершения — только для задач
  download-to-SD (`STORAGE_FILE_DISK`).
- Опция «качать на SD, потом установить» (2× место) — осознанный выбор
  пользователя ради рейтинга; UI должен это объяснять.
- Синергия с F-A (`SYSMODULE_PLAN.md`): sysmodule сидирует download-to-SD
  задачи в фоне, пока пользователь играет — единственный путь к
  долгому сид-тайму на консоли.

## 1. Целевая модель

1. **База (эпик 4.4):** приём и обслуживание запросов пиров, учёт
   отданного, честный announce, режим «Сидирование» после завершения
   (опция + лимиты: целевой ratio, время, потолок скорости отдачи).
2. **F-E:** choker BEP 3 — 4 unchoke-слота, пересчёт каждые 10 с;
   при загрузке слоты отдаются пирам с максимальной скоростью **к нам**,
   при сиде — с максимальной скоростью приёма **от нас**; optimistic
   unchoke с ротацией 30 с; anti-snubbing (пир, не приславший блок 60 с
   при наших pending-запросах, помечается snubbed и слот не занимает).
3. **Super-seeding (BEP 16)** — опциональная последняя фаза (режим
   «единственный сид»: редкая ситуация, низкий приоритет).

## 2. Дизайн

### 2.1 Приём запросов (`peer.c`)

- Новые сообщения: `case MSG_REQUEST` и `MSG_CANCEL` (id 8; константы
  дополнить рядом с `peer.h:13–20`).
- Валидация: пир должен быть нами unchoked; length ≤ 16 КиБ
  (`BLOCK_SIZE`, `piece.h:7`) с допуском до 32 КиБ; кусок у нас есть
  (`have_bf`); иначе — игнор/дисконнект по счётчику нарушений.
- Очередь входящих запросов per peer: фиксированное кольцо (например 64
  записи ~ 1 МиБ отдачи впереди); переполнение → choke пира.
- `MSG_CANCEL` удаляет запись из кольца (endgame-этикет).

### 2.2 Путь отдачи (`torrent.c` + `peer.c`)

- В `torrent_tick()`: для unchoked-пиров с непустой очередью — прочитать
  блок (`storage_read`), собрать `MSG_PIECE` и отдать в send-путь.
- Send-путь: `sbuf` рассчитан ровно на одно piece-сообщение
  (`PEER_BUF_SIZE`, `peer.h:9`) — блок 16 КиБ помещается; правило: новый
  блок не сериализуем, пока `sbuf` не опустошён (`net_send` уже
  поддерживает частичную запись и докладку хвоста, `net.h:52–56`).
  Итог: естественный backpressure без новых буферов — RAM-нейтрально
  (важно для F-A-бюджета).
- Broadcast `MSG_HAVE` по завершении куска всем пирам (аудит фазы 0:
  сейчас, судя по отсутствию отправки have в `peer.c`, не делается —
  проверить; have нужен и личу: пиры пересчитывают interest).
- Троттлинг: глобальный потолок отдачи (по умолчанию ~1 МиБ/с,
  настройка); при активном install-конвейере отдача дополнительно
  придушивается (SD-чтение конкурирует с записью — см. риски).

### 2.3 Учёт и announce (`tracker.c`, `torrent.c`)

- `t->uploaded_bytes` per torrent; в `torrent_stat_t` — `uploaded`,
  `upload_bps` (расширение `torrent.h:23–37`).
- `http_announce_once` (`tracker.c:81`): параметр `uploaded` реальный;
  `event` как аргумент: `started` только на первом анонсе, `completed`
  при завершении, `stopped` при паузе/удалении/выходе, переанонсы — без
  event. UDP-путь (`udp_announce`, `tracker.c:259`) — те же поля в пакете
  BEP 15 (аудит: сейчас там нули).
- `upload_only` в extended handshake (`peer.c:123`) — выставлять честно:
  0 всегда после этого плана (мы отдаём), 1 — только если отдача
  отключена настройкой.
- Аудит peer_id: зафиксировать Azureus-style префикс (`-PX0500-` +
  random) и консистентный User-Agent (`tracker.c:112`, сейчас
  `pipensx/0.4`) — стабильная идентификация клиента для статистики
  трекера.
- Персист `uploaded` per task в `queue.bencode`
  (`download_manager.cpp:1404–1410`) — ratio в UI переживает рестарты
  (на сам трекер уходит сессионный кумулятив, как положено).

### 2.4 Choker (новый `src/core/choker.c`)

- Вход: скорости per peer (rx уже меряется — телеметрия
  `rx_bps` в `torrent.c:463`; добавить симметричный tx_bps).
- Каждые 10 с: сортировка interested-пиров по метрике режима → топ-4
  unchoke; каждые 30 с один из choked+interested — optimistic (ротация,
  исключая snubbed).
- Anti-snubbing: `last_block_received_ms` per peer; > 60 с при непустом
  pipeline → snubbed: choke, запросы к нему только в endgame.
- Snubbed/choke-решения — только внутри tick (движок однопоточный,
  `torrent.h:64`) — блокировок не нужно.

### 2.5 Режим сидирования (`download_manager.cpp`, UI)

- Новый статус `Seeding` после `Complete` для download-to-SD задач
  (persist-словарь статусов: `download_manager.cpp:1108–1135`).
- Настройки: тумблер «Сидировать после загрузки», целевой ratio
  (по умолчанию 1.0), максимум времени, потолок скорости.
- Карточка задачи: отдано / ratio / скорость отдачи.
- Стоп-условия шлют `event=stopped`.

## 3. Фазы

### Фаза 0 — протокольная честность (S)
Announce-события (started/completed/stopped), реальный `uploaded` в
HTTP+UDP анонсах (пока фактический 0 — но канал репорта корректный),
peer_id/UA-аудит, `MSG_HAVE`-broadcast, честный `upload_only`.
**Уже одна эта фаза чинит учёт pipensx на RuTracker.** Кода мало,
эффект максимальный.

### Фаза 1 — отдача, база 4.4 (M)
`MSG_REQUEST`/`MSG_CANCEL`, кольцо запросов, отдача блоков с
backpressure, учёт uploaded, простая политика unchoke (все interested до
N=8 слотов), глобальный потолок скорости. Тест: loopback-рой из двух
инстансов движка на PC (паттерн `tests/test_torrent.c`) — полный цикл
сид→лич.

### Фаза 2 — choker F-E (M)
4 слота по скорости, optimistic 30 с, anti-snubbing, tx_bps-телеметрия,
режим Seeding в DownloadManager + UI + лимиты. Unit-тесты choker'а с
детерминированными скоростями.

### Фаза 3 — интеграция и фон (S/M)
Телеметрия ratio в лог/диагностику; согласование с F-A: сидирование в
sysmodule (отдача RAM-нейтральна — вписывается в бюджет
`SYSMODULE_PLAN.md` §2), лимиты по батарее/типу задачи.

### Фаза 4 — super-seeding BEP 16 (M, опционально)
Только по фактической потребности (мы — первичный сид раздачи). По
умолчанию не делать.

## 4. Риски

| Риск | Вер-ть | Митигация |
|---|---|---|
| SD-чтение отдачи конкурирует с записью install-конвейера (стойки гейта, PERF 7.1) | средняя | приоритет установke: жёсткий троттлинг отдачи при активном install; отдача из piece-буферов в RAM, пока кусок ещё не вытеснен |
| Слабый WiFi-uplink Switch: отдача душит собственную закачку | средняя | потолок отдачи по умолчанию ~1 МиБ/с; choker и так отдаёт только 4–5 пирам |
| Некорректные/злонамеренные request'ы (OOM, чтение мимо диапазона) | средняя | строгая валидация §2.1, счётчик нарушений → disconnect; fuzz-кейс на парсер request |
| Ошибка в событиях announce (двойной completed и т.п.) искажает статистику на трекере | низкая | unit-тест конечного автомата событий; проверка на локальном opentracker |
| Ожидание «ratio растёт от каждой игры» не сбудется (stream-install несидируем, §0.3) | факт | честный UI-текст; опция download-to-SD «ради рейтинга»; синергия с F-A |

## 5. Критерии приёмки

1. Loopback-тест на PC: второй инстанс докачивает торрент целиком только
   от первого; uploaded первого == размеру торрента ± служебка.
2. Announce-лог на реальной раздаче RuTracker: `started` один раз,
   переанонсы без event, `completed` один раз, `stopped` при выходе;
   `uploaded` монотонно растёт. Рейтинг в профиле отражает сессию.
3. Тонкий рой (≤5 пиров): скорость загрузки с включённой отдачей ≥
   базлайна без отдачи (ожидаемо выше за счёт взаимности).
4. Отдача при активной stream-установке не замедляет установку >5%
   (телеметрия `sink_bps`/`processed_bps` из PERF-плана).
5. Fuzz-прогон парсера MSG_REQUEST/MSG_CANCEL без падений.

---

## 6. Не-цели

- Сидирование содержимого, установленного stream-install (данных нет —
  §0.3).
- Long-term seed-бокс поведение (авторестарт раздач после переустановки
  и т.п.).
- BEP 16 по умолчанию (только опционально, фаза 4).
- MSE/uTP — отдельный эпик 4.5 (усиливает и отдачу, но не входит сюда).

## 7. Оценка

Фаза 0 — S; фаза 1 — M; фаза 2 — M; фаза 3 — S/M; фаза 4 — M (опц.).
Суммарно M/L, что согласуется с оценкой F-E (M) + части 4.4 (M) в
`IMPROVEMENT_PLAN.md`. Порядок «0 → 1» имеет смысл выпускать сразу —
ценность для RuTracker-аккаунтов не ждёт choker'а.

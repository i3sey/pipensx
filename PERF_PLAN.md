# План оптимизации тракта «загрузка → установка NSP/NSZ»

Дата: 2026-07-02. Цель — максимальная скорость скачивания и установки игр (NSP/NSZ)
на Switch. Все ботлнеки подтверждены по коду; ссылки — `файл:строка`.
Это план по производительности, не security-аудит.

## Фаза 0 — базлайн (до любых правок)

Телеметрия уже есть (`sink_bps`, `processed_bps`, `zstd_us/aes_us/sha`,
`ncm_stall`, `buffer_stall`, `ncm write_us`). Зафиксировать эталонный прогон:
одна и та же игра (~10 ГБ NSZ), один swarm, замер общего времени + разбивка по
стадиям. Каждая фаза ниже принимается только по дельте к базлайну.

## Замер 2026-07-02 (pipensx.log, после фаз 1–2 и 3.1–3.2)

Частичный прогон: Zelda Link's Awakening NSZ 2.06 ГБ (985 кусков × 2 МиБ),
прерван пользователем на 451/985 (~178 с). Что показал:

- **Установка**: 1.67 ГБ за 173.6 с → 9.6 МБ/с средняя. `writer_us` = 106.8 с
  из 174 — 61% времени install-воркера уходит в ncm-запись. Крипто после 2.1
  пренебрежимо: zstd 6.6 с + AES 1.0 с + SHA 2.3 с ≈ 10 с суммарно —
  ожидание фазы 2 подтверждено, SHA/AES больше не узкое место.
- **Потолок ncm-записи ~16–17 МБ/с**: 94% duty при 15.7 МБ/с; вызов 1 МиБ
  в среднем ~57 мс, столлы 128–144 мс. После починки сети это станет стеной.
- **Скачивание — главный тормоз**: ~5 МБ/с при 3 активных пирах бóльшую часть
  прогона. Трекер дал 30 пиров, живых 3. На ~140-й секунде пришла пачка
  (`peer_queue=196`), active 3→7 — скорость выросла до 12.5 МБ/с.
- **`meta trackers=1`**: magnet-фаза опрашивала 4 трекера (bt/bt2/bt3/bt4
  t-ru.org), в торрент попал один. Ре-аннонса до ~140 с не было.
- **`pipeline=64` пришпилен** у каждого активного пира весь прогон
  (64 × 16 КиБ = 1 МиБ in-flight), rx на пира 1–2 МБ/с и никогда выше —
  request_limit и есть потолок на пира.
- **Head-of-line stall 10 с**: один пир (rx 91 КБ/с) держал head-окно,
  `oldest_request_ms=9953`, хедж сработал только по порогу
  `hedge_after_ms=5000`. Затем backlog слился лавиной: ~180 МБ (куски 358–445)
  за ~2 с упали в буфер.
- **Backpressure блокирует event loop**: после лавины буфер упёрся в лимит
  134 МиБ, писатель ждал до 4.8 с (`buffer_stall wait_us=4826332`,
  `piece_cb_max_us=724732`, busy 268‰) — торрент-цикл стоял, rx упал
  12.5 → 4.8 МБ/с, пайплайны пиров обнулились.
- 3 × `set TCP receive buffer: No buffer space available` во время
  connect-волн (~12 одновременных сокетов) — пул сокет-буферов исчерпан,
  SO_RCVBUF из 1.4 применяется не всегда.
- Мелочи: `last_piece_age_ms` печатает epoch-мусор для пира без единого
  piece; image relay качает 4 МиБ обложки по сети посреди загрузки.

Выводы разнесены по пунктам: 1.6 (новый), 3.3 (приоритет понижен),
3.5 (новый), 5.1–5.2 (подтверждены замером), 5.3 (новый), фаза 6 (новая).

## Фаза 1 — сеть (дёшево, самый большой выигрыш по download)

### 1.1. Починить `net_send` — неблокирующая отправка через sbuf ✅
Реализовано: `net_send` возвращает число отправленных байт (`EAGAIN` — не
ошибка), непринятый ядром хвост уходит в `sbuf` пира (`peer_send_raw`), при
`sbuf_len > 0` торрент-цикл подписывается на `POLLOUT` и досылает через
`peer_flush`. `send_msg` резервирует место под весь фрейм заранее — частично
поставленный в очередь фрейм невозможен. Покрыто тестом
(`test_partial_send_queues_tail_and_flush_completes_frame`).

### 1.2. Пить сокет досуха ✅
Реализовано: `peer_recv` чередует разбор накопленных сообщений с `recv` до
`EAGAIN`, поэтому одно poll-событие вычитывает всё доступное. `rbuf` отделён от
send-буфера и вмещает payload 256 KiB плюс 4-байтовый length prefix. Для 96
пиров это около 24 MiB — приемлемо для поддерживаемого application mode.

### 1.3. Кольцевой буфер вместо memmove ✅
Реализовано: rbuf получил read-курсор (`rbuf_head`). Consume сообщения двигает
курсор вместо memmove хвоста; при полном сливе буфера курсор сбрасывается в нуль
бесплатно. memmove-компакция осталась только на редкий случай, когда хвост
заполнен, а перед курсором есть уже прочитанный префикс (`peer_recv`). Курсор
сохраняет непрерывность payload в rbuf — указатели `on_block` не ломаются.
Покрыто тестом (`test_recv_reassembles_message_split_across_reads`).

### 1.4. Зафиксировать TCP receive buffer ✅
В поддерживаемом application mode Borealis уже использует `sb_efficiency = 8`.
Реализовано: `net_tcp_connect` явно запрашивает `SO_RCVBUF = 256 KiB` для всех
пировых и metadata-сокетов; невозможность применить настройку не рвёт соединение.

### 1.5. Убрать блокировки event loop ✅
Реализовано: tracker announce вынесен на worker-поток (`announce_worker`) —
event loop запускает прогон (`announce_start`), продолжает обслуживать пиров и
забирает результат на следующем тике (`announce_collect`), не блокируясь.
Первичный и ре-аннонс оба async; при недоступности потоков — синхронный fallback.
UPnP discover+mapping — one-shot фоновый поток при старте (`upnp_worker`, Switch),
joined в `torrent_destroy`. Поток аннонса тоже joined в destroy (ограничен
curl-таймаутом, не влияет на loop во время загрузки).

### 1.6. Все трекеры из magnet + ранний ре-аннонс при дефиците пиров ✅
Проблема по замеру 2026-07-02: в метаданные торрента попадал один трекер
(`meta trackers=1`), хотя magnet-фаза успешно опрашивала четыре. Пирами из
остальных трёх торрент не пользовался. Замер показал: рост active 3→7 удвоил
скорость (5 → 12.5 МБ/с) — самый дешёвый и большой выигрыш по download.

Реализовано:
- `MagnetResolver::buildTorrent` вшивает в торрент `announce-list` со всеми
  RuTracker-миррорами (`rutrackerTrackerCandidates`), а не единственный `tr=`
  из magnet (`src/app/magnet_resolver.cpp`). `metainfo_parse` уже разворачивает
  все tier'ы в `mi->trackers[]`, `tracker_announce` аннонсит по всем. Тест
  `test_catalog` теперь печатает `trackers=4` вместо 1.
- Ранний ре-аннонс при дефиците пиров: `torrent_tick` при
  `active < TRACKER_STARVED_ACTIVE_PEERS` (5) не ждёт штатных 30 мин, а
  ре-аннонсит, начиная с `TRACKER_STARVED_REANNOUNCE_MS` (15 с)
  (`src/core/torrent.c`). Окно-гейт одновременно rate-limit'ит трекер и держит
  подсчёт активных пиров вне hot-path каждого тика.
- DHT-результаты уже вливаются в peer_queue по мере прихода
  (`cb_dht_peer` → `queue_push`, поиск заведён в `torrent_create_ex`, тики DHT
  по интервалу и по POLLIN) — изменений не потребовалось.
- `numwant` не трогали: он уже `=200` в HTTP-аннонсе и упирается в 200-элемент
  буферы (`compact[200*6]`) по всему тракту — поднимать = согласованная правка
  ≥5 мест ради пиров, которые некуда положить (peer_queue и так 1024).

## Фаза 2 — аппаратное крипто (средне по усилию, большой выигрыш по install)

### 2.1. Заменить программное крипто на libnx crypto extensions ✅
Реализовано: на Switch всё крипто идёт через CE-ускоренные примитивы libnx.

- SHA-1: `sha1_ctx_t` на `__SWITCH__` — это libnx `Sha1Context`, функции
  `sha1_*` — тонкие обёртки над `sha1Context*`/`sha1CalculateHash`
  (`src/core/sha1.h`, `src/core/sha1.c`); софт-реализация осталась для PC.
- SHA-256 каждого NCA: mbedtls заменён на `Sha256Context`/`sha256Context*`
  (`src/install/install_backend_switch.cpp`).
- AES-CTR NSZ-секций: mbedtls заменён на `Aes128CtrContext`/`aes128CtrCrypt`;
  внутриблочное смещение обрабатывается сжиганием первых `offset % 16` байт
  keystream (`src/install/package_stream.cpp`, `AesCtr::transform`).
  PC-путь (OpenSSL) не тронут — тесты `test_package_stream` покрывают
  эквивалентную логику счётчика.

Ожидание: SHA ~5–10×, AES-CTR ~5× в критическом пути NSZ
(zstd → AES → SHA → ncm). Подтвердить эталонным замером из фазы 0.

## Фаза 3 — конвейер установки

### 3.1. Убрать O(n) erase-from-front ✅
Реализовано: общий `ByteQueue` (append в хвост, consume курсором `head_` за O(1))
заменил `std::vector<uint8_t> pending_` в `PackageStream::Impl` и `NczDecoder`.
Consume больше не memmove'ит хвост на каждой границе PFS0/NCZ; прочитанный
префикс освобождается лениво в `append` (O(1) при полном сливе, иначе один сдвиг
только непрочитанного остатка). Покрыто существующим тестом с рваными чанками
(1/7/31/4093/17/65537 байт) через все PFS0/NCZ/block-пути.

### 3.2. Увеличить выходной чанк в ncm ✅
Реализовано: `outputBuffer_` поднят с 256 KiB до 1 MiB
(`src/install/package_stream.cpp:678-679`) — каждый
`ncmContentStorageWritePlaceHolder` IPC-вызов теперь ≤1 MiB вместо ≤256 KiB
(solid-режим; block-режим уже писал по `blockSize_`, не затронут). Замер
`ncm_stall`/`write_us` (порог 100 мс, `src/install/install_backend_switch.cpp:444`)
— на реальном железе.

### 3.3. Распараллелить стадии install — приоритет понижен
Один `installWorker_` (`src/app/download_manager.cpp:222`) последовательно
гонит zstd → AES → SHA → ncm. Исходная мотивация — спрятать крипто за ncm —
устарела: по замеру 2026-07-02 zstd+AES+SHA ≈ 10 с из 174 с, т.е. параллелизм
даст ≤6%. Стена — сама ncm-запись (61% времени воркера). Делать только после
3.5 и повторного замера, если ncm-потолок поднять не удастся.

### 3.4. Не тащить delta-фрагменты апдейтов ✅ (вариант b + ранний cnmt)
Было: delta-NCA скачивались полностью, стримились сквозь весь конвейер
(zstd → AES → SHA → ncm-плейсхолдер) и дропались при commit.

Реализовано:
- Ранний разбор cnmt: как только meta-NCA (`.cnmt.nca`) дописан, backend
  регистрирует его в NCM и парсит cnmt (`prepareEarlyMeta`,
  `src/install/install_backend_switch.cpp`) — `deltaIds` известны ещё во время
  стрима, а не при commit. Best-effort: сбой раннего разбора не фатален,
  commit повторяет разбор штатно; commit переиспользует ранний результат
  (без второго чтения cnmt).
- Скип на уровне PFS0-entry: `PackageCallbacks::skipFile` — стрим спрашивает
  backend (`InstallBackend::shouldSkipFile`) перед каждым entry; delta-entry
  потребляется сырым счётчиком байт: без NCZ-декода, без zstd, без AES, без
  SHA, без ncm-плейсхолдера и записи (`src/install/package_stream.cpp`).
  Телеметрия: событие `package event=skip name= bytes=`.
- Delta, идущие в PFS0 *до* cnmt, стримятся как раньше и дропаются при
  commit (плейсхолдеры удаляются) — поведение не менялось.
- Тесты: `testSkipFileDropsEntryWithoutCallbacks` (entry выпадает без
  callbacks, соседние файлы целы), `testSkippedNczEntryBypassesDecoder`
  (скип-entry с мусором вместо NCZ — декодер не трогается).

Вариант (a) — исключение delta из *скачивания* — не реализован: требует дырок
в строгом порядке стрима (skip-диапазоны в piece manager + storage,
fast-forward `producerOffset_`/`consumed()`), а выигрыш ограничен: к моменту
разбора cnmt запросы уже выданы на `requestAheadBytes_` (192 МиБ) вперёд.
Экономия сети возможна только для delta дальше этого окна — отдельная задача,
если замер покажет смысл. Выигрыш (b): delta не занимают ncm-запись (стена
по замеру: 61% времени install-воркера) и крипто; на апдейтах это гигабайты.

### 3.5. Поднять потолок ncm-записи (~16–17 МБ/с по замеру) ✅
Реализовано: выходной буфер solid NCZ увеличен с 1 до 4 МиБ, а regression-тест
проверяет, что writer действительно получает 4-МиБ чанк. Идентификатор целевого
хранилища централизован, все события `ncm`/`ncm_stall` теперь содержат
`target=sd`, включая interval/summary, setup, stall, commit и rollback.

Текущий Switch-backend поддерживает установку только на SD
(`NcmStorageId_SdCard`), поэтому честный замер NAND в этой реализации невозможен
без отдельной функции выбора места установки. Телеметрия теперь однозначно
разделяет цель и готова для сравнения `ncm_us`/`write_us`, когда появится NAND-path;
фактический потолок 4-МиБ чанка проверяется только замером на устройстве.

## Фаза 4 — системные ручки (очень дёшево)

### 4.1. CPU boost на время загрузки/установки
Реализовано через `appletSetCpuBoostMode(ApmCpuBoostMode_FastLoad)`.
1785 МГц vs 1020 — прямое ускорение zstd/AES/SHA на ~75%. Включать при активной
задаче, выключать в idle.

### 4.2. Запрет автосна
Реализовано: `appletSetAutoSleepDisabled(true)` включается на время активной
задачи, а сохранённое исходное значение возвращается после завершения.

### 4.3. Адаптивный RAM-бюджет ✅
Реализовано: на Switch доступная память определяется внутри уже выделенного
newlib heap: свободный хвост `fake_heap_end - sbrk(0)` складывается с
`mallinfo().fordblks`. Разность `InfoType_TotalMemorySize` и
`InfoType_UsedMemorySize` после `svcSetHeapSize` показывает только остаток вне
heap (на реальном запуске около 4 MiB), поэтому сохраняется лишь как
диагностический `kernel_headroom` и не блокирует установку. Предпочтительный
резерв 256 MiB под UI, метаданные и системные сервисы уменьшается до половины
доступной heap-памяти (но не ниже 64 MiB). Оставшийся рабочий бюджет (не более
384 MiB) адаптивно делится между reorder-буфером, очередью и lookahead с учётом
размера куска; при сильном дефиците reorder-буфер уменьшается ниже
предпочтительных 64 MiB. Потоковая установка отклоняется только когда после
минимального резерва не помещаются буфер и хотя бы один lookahead-кусок. При
недоступности heap-замера применяется консервативный fallback. Событие
`ram_budget` фиксирует источник, kernel headroom, доступную heap-память, резерв,
размер куска, настроенный пик и все рассчитанные лимиты; тесты покрывают
реальный 4 MiB kernel-headroom, низкий RAM-сценарий и пиковые инварианты.

## Фаза 5 — стратегия скачивания (после фаз 1–3)

### 5.1. Адаптивное окно строгого порядка + агрессивный хедж ✅
Проблема по замеру 2026-07-02: `strict_order_lookahead = 32` фикс — 
sequential-загрузка режет эффективность swarm; медленный пир держал head-окно
10 с (`oldest_request_ms=9953`), хедж по статическому `hedge_after_ms=5000`
проснулся слишком поздно, после чего backlog ~180 МБ слился лавиной и
переполнил буфер (см. 5.3).

Реализовано:
- Адаптивное окно: AIMD-контроллер в `PackageCoordinator::adaptiveLookahead`
  (`src/app/download_manager.cpp`) по состоянию install-буфера — stall
  (после 5.3 — пауза request gate) или буфер >75% лимита → окно /2;
  буфер <50% → окно +4; между порогами окно держится (гистерезис против
  осцилляций). Clamp [8, 64] кусков, старт 32, шаг раз/сек. Цикл менеджера
  применяет окно через новый API `torrent_set_strict_lookahead`
  (`src/core/torrent.c`) после каждого `torrent_tick`.
- Адаптивный хедж-порог: per-peer EMA латентности блока request→piece
  (`block_lat_ema_ms`, `src/core/peer.c`, сэмпл в MSG_PIECE только для
  реально запрошенных блоков), `adaptive_hedge_after_ms` (`src/core/torrent.c`)
  берёт медиану EMA активных пиров × 4, clamp [500 мс, статический
  `hedge_after_ms`=5000]. Меньше 2 сэмплированных пиров — статический порог
  (хеджировать всё равно не к кому). Считается раз в HEDGE_INTERVAL_MS (250 мс)
  внутри `schedule_hedged_requests`.
- Телеметрия: `lookahead=` и `hedge_ms=` в 5-секундной строке `torrent`.
- Тесты: `test_adaptive_hedge_follows_median_latency` (медиана, клампы,
  фолбэк), `test_piece_receipt_samples_block_latency` (EMA, unsolicited-блоки
  не сэмплируются).

### 5.2. Динамический pipeline на пира ✅
Проблема по замеру 2026-07-02: у каждого активного пира `pipeline=64`
(1 МиБ in-flight) пришпилен весь прогон при rx 1–2 МБ/с — очередь всегда
полная, константа `request_pipeline_limit` и есть потолок на пира.

Реализовано:
- Per-peer EMA скорости скачивания: `dl_rate_bps` в `peer_t` (`src/core/peer.h`),
  сэмплится раз/сек в `torrent_tick` из монотонного `p->downloaded` через
  `ema_update` — телеметрия для этого не нужна.
- `peer_pipeline_limit` (`src/core/torrent.c`) считает окно как
  ~`PIPELINE_TARGET_MS` (2 с) данных на измеренной скорости пира:
  `want = dl_rate_bps * 2000/1000 / BLOCK_SIZE`, clamp `[MIN_REQUEST_PIPELINE,
  ceiling]`, поверх — прежний geometric backoff по `timeout_strikes`. Быстрый
  пир получает глубокую очередь до потолка, медленный держит лишь то, что
  успевает отработать. До первого сэмпла — bootstrap `PIPELINE_BOOTSTRAP_BPS`
  (1 МиБ/с), чтобы новый пир сразу качал.
- Потолок in-flight в stream-install поднят 64 → 256 (4 МиБ = MAX_PIPELINE,
  `src/app/download_manager.cpp`) — теперь адаптив может дать быстрому пиру до
  4 МиБ, а не упираться в 1 МиБ. Буфер `pipeline[MAX_PIPELINE=256]` уже был
  такого размера — доп. памяти нет. Строгий порядок + lookahead 32 куска
  ограничивают суммарный in-flight, так что один пир окно не выедает.
- DownloadOnly (options=NULL) уже имел ceiling MAX_PIPELINE — адаптив
  распространяется и на него.

### 5.3. Backpressure без блокировки event loop ✅
Проблема по замеру 2026-07-02: при заполнении буфера установки (лимит 134 МиБ)
writer ждал места прямо в piece-callback — `buffer_stall wait_us` до 4.8 с,
`piece_cb_max_us=724732`, busy 268‰. Весь торрент-цикл стоял: rx падал
12.5 → 4.8 МБ/с, пайплайны пиров обнулялись, таймауты тикали.

Реализовано:
- Блокирующий `waitForBufferSpaceLocked` удалён из sink-пути целиком (вместе
  с condvar `queueSpace_`): piece-callback больше никогда не ждёт — чанк
  принимается всегда, цикл продолжает обслуживать handshake/keepalive/POLLOUT.
- Вместо ожидания — request gate (`updateRequestGateLocked`,
  `src/app/download_manager.cpp`): буфер достиг лимита → `canRequestPiece`
  возвращает false для всех кусков (движок перестаёт выдавать новые request'ы
  через существующий `request_allowed`-хук в `piece_mgr_pick`), уже летящие
  блоки доезжают. Гистерезис: пауза на 100% лимита, резюм на 75% — гейт не
  дребезжит. Перелёт сверх лимита ограничен окном строгого порядка +
  `requestAheadBytes_`-гейтом (новые запросы за лимитом не выдаются).
- Пауза гейта считается stall-событием для AIMD-окна 5.1 — окно сужается,
  снижая вероятность следующей лавины.
- Телеметрия: в строке `buffer` вместо `waits/wait_total_us/wait_max_us` —
  `pauses/paused_ms/pause_max_ms/gate_paused`; события `request_gate`
  (pause/resume) с длительностью. `tools/analyze_throughput_log.py` читает
  новые поля и старые (fold в pauses) для старых логов.
- Тесты: analyzer — детект давления по `pauses` и по legacy `waits`; движковый
  гейт покрыт существующим `test_strict_order_stops_at_request_gate`.

## Фаза 6 — гигиена по логу 2026-07-02 (дёшево)

### 6.1. SO_RCVBUF: не хватает пула при connect-волнах ✅
Реализовано: запрос 256-КиБ `SO_RCVBUF` перенесён из `net_tcp_connect` на момент
после успешной проверки BitTorrent-handshake — и в основном torrent-пути, и в
magnet resolver. Неуспешные и зависшие connect-сокеты теперь используют
системный буфер и не расходуют увеличенный пул; ошибка настройки после handshake
остаётся безопасным fallback и не рвёт соединение. Regression-тесты проверяют,
что connect не увеличивает буфер заранее, helper применяет 256 КиБ, а валидный
handshake вызывает его. Влияние fallback на скорость остаётся device-only
замером на реальном Switch.

### 6.2. Телеметрия: мусор в `last_piece_age_ms` + цель установки ✅
Реализовано: peer-телеметрия печатает signed `last_piece_age_ms=-1`, если пир
ещё не прислал ни одного piece или метка времени некорректно находится в будущем;
для валидной метки сохраняется фактический возраст. Поведение покрыто unit-тестом.
Цель записи уже добавлена пунктом 3.5: все события `ncm`/`ncm_stall` содержат
`target=sd`, соответствующий `NcmStorageId_SdCard` текущего backend.

### 6.3. Пауза image relay при активной задаче ✅
Реализовано: главный цикл передаёт `DownloadManager::hasActiveTransfer()` в
`GameMetadataService`. Пока задача активна, image worker продолжает обслуживать
memory/disk/local cache, но cache-miss не запускает HTTP/relay: URL и callbacks
остаются в очереди до снятия паузы. Возобновление будит workers немедленно;
уже начавшийся до паузы HTTP-запрос не прерывается. Regression-тесты подтверждают
доступность disk cache во время паузы и отсутствие callback до её снятия для
сетевого cache-miss.

× 2 МиБ) + апдейт 0.04 ГБ. Скачано и установлено целиком: ~450 с от

### Что подтвердилось (стало лучше)

- **lookahead застрял на 8** (16 МиБ окна) с ~285 по ~420 с — буфер держался

  handshake (6.1 сузила проблему, но пул всё ещё исчерпывается).
- **3.4 не сработал на апдейте**: cnmt-парс дал `deltas=4`, но ни одного

физическая скорость ncm-записи; второй резерв — сглаживание сопряжения

## Фаза 7 — сопряжение сеть↔установка (по замеру 2026-07-03)

### 7.1. Rate-matched request gate вместо бинарного ✅
Проблема: бинарный гейт (пауза на 100%, резюм на 75%) давал окна rx=0 по
20–28 с; жёсткий `requestAheadBytes_`-потолок — третье мёртвое окно.

Реализовано: гейт вынесен в отдельный модуль `pipensx::RequestGate`
(`src/app/request_gate.{hpp,cpp}`), три состояния вместо двух:
- **Free** (буфер <75% лимита): свободный поток, admission edge =
  `producerOffset + requestAheadBytes` — поведение как раньше. Плюс
  сглаживание жёсткого потолка: при исчерпанном окне edge продолжает
  ползти со скоростью drain'а (overshoot ограничен 4 кусками) — вместо
  полного стопа до скачка `producerOffset_`.
- **Throttled** (75–100%): токен-бакет — edge якорится к фронтиру
  прибывших байт (+2 куска burst) и растёт со скоростью измеренного
  drain'а (EMA processed-байтов install-воркера, окно 500 мс, в сетевых
  байтах). Скачивание продолжается со скоростью потребления вместо rx=0.
  Накопленный free-flow-кредит при входе в троттл сгорает — лавины нет.
- **Paused** (≥100%): аварийный потолок, как раньше, но резюм на
  `лимит − кусок` (не 75%): дальше поток ведёт токен-бакет, дренаж
  64 МиБ при rx=0 больше не нужен. Токены во время паузы не копятся.

Floor `producerOffset + кусок` держит head-кусок запрашиваемым даже при
нулевом drain — дедлок гейта на самом себе невозможен. Рефилл получает
heartbeat из `adaptiveLookahead()` (каждый tick менеджера), так что edge
растёт и во время rx-затишья. Телеметрия: в строке `buffer` добавлены
`throttles/throttled_ms/gate_throttled/drain_bps`; события `request_gate`
— `throttle`/`throttle_end`, `resume` дополнен `throttled=`. Анализатор
читает новые поля (fallback для старых логов сохранён).

Тесты: `tests/test_request_gate.cpp` — free flow, аварийная пауза с
быстрым резюмом, якорение к фронтиру, рост edge по drain-rate (~64 МиБ за
4 с при 16 МБ/с), нулевой drain не дедлочит head, выход из троттла,
creep за исчерпанным окном, rollover пакета, отсутствие банка токенов в
паузе; analyzer — детект давления по `throttles` без `pauses`.
Ожидание (проверить замером): минус ~50–70 с на прогоне (449 → ~385–395 с),
плавный rx без обвалов в ноль.

### 7.2. Не остужать пиров во время паузы/троттлинга гейта ✅
Проблема: после резюма rx восстанавливался ~30 с — `dl_rate_bps` EMA пиров
затухла за паузу → pipeline съёжился до bootstrap, окно AIMD на минимуме.

Реализовано:
- Движок: `torrent_set_rate_freeze` (`src/core/torrent.{h,c}`) — пока
  флаг взведён, per-peer сэмплинг (`sample_peer_rates`, выделен из
  `torrent_tick`) не трогает `dl_rate_bps` EMA и сбрасывает базу
  `rate_last_downloaded`: замороженные интервалы не входят в замер вовсе,
  первый сэмпл после резюма меряет только пост-резюмные байты.
  `peer_pipeline_limit` считает глубину от сохранённой EMA — pipeline
  возвращается к до-гейтовой глубине сразу после резюма.
- Координатор: `requestsCurtailed()` — гейт 7.1 не в Free (Throttled или
  Paused: измеренный rx отражает гейт, а не пира); цикл менеджера каждый
  tick транслирует это в `torrent_set_rate_freeze` рядом с
  `torrent_set_strict_lookahead`.
- AIMD-окно (5.1) заморожено на время Paused: `adaptiveLookahead`
  возвращает текущее окно без адаптации — буфер в паузе ничего не говорит
  о swarm'е, окно не перемалывается до минимума серией халвингов.
  Одиночный stall-event от перехода в паузу сохранён (одно /2 после
  резюма, как в 5.1). В Throttled AIMD работает как раньше — гейт
  запросы выдаёт; floor/recovery окна — отдельный пункт 7.3.

Тест: `test_rate_freeze_preserves_peer_dl_rate` (`tests/test_torrent.c`) —
незамороженный idle-интервал затухает, замороженный держит EMA точно и
двигает базу, после резюма здоровый интервал поднимает EMA без учёта
затишья. DownloadOnly-путь не затронут (freeze ставится только при
stream install).

### 7.3. Floor lookahead и recovery после гейта ✅
Проблема: `lookahead=8` (16 МиБ) при 9–11 живых пирах — слишком узко,
inflight падал до единиц блоков; после резюма гейта окно отрастало по
+4/с от минимума.

Реализовано:
- Динамический floor окна: `torrent_stat` отдаёт `num_active_peers`
  (пиры в PS_ACTIVE, в отличие от `num_peers` — занятых слотов), цикл
  менеджера передаёт его в `adaptiveLookahead(activePeers)`. Нижний clamp
  AIMD = `max(lookaheadMin, min(lookaheadMax, active*2))` — каждому живому
  пиру достаётся ≥2 куска окна, устойчивое давление буфера больше не
  сжимает окно до голодных единиц inflight.
- Recovery после паузы: `adaptiveLookahead` запоминает
  `lookaheadHealthy_` — окно на адаптационных тиках без stall при гейте
  в Free. Переход Paused→Throttled/Free восстанавливает окно к healthy
  сразу (`updateRequestGateLocked`), pending stall-event от паузы делит
  его один раз по семантике 5.1 — резюм стартует с ~healthy/2 с floor'ом
  по swarm'у, а не с минимума.

Тест: `test_stat_counts_active_peers` (`tests/test_torrent.c`) —
`num_active_peers` считает только PS_ACTIVE. Поведение floor/recovery —
device-замер (фаза 7, шаг 1: ожидание −50–70 с на прогоне вместе с
7.1/7.2).

### 7.4. Потолок ncm-записи: последняя попытка поднять ✅ (код; вердикт — замер на устройстве)
Чанк 1→4 МиБ потолок не сдвинул ⇒ дело в SD/FS-пути. Главный рычаг — сменить
цель записи с SD на NAND (eMMC обычно быстрее SD).

Реализовано — выбор цели установки:
- `InstallStorageTarget { SdCard, Nand }` (`src/install/install_backend.hpp`);
  `createInstallBackend(root, target)` (default SD). Switch-backend маппит
  target → `NcmStorageId_SdCard`/`NcmStorageId_BuiltInUser`, строку `target=`
  (`sd`/`nand`) и storage id в application record (`ncmStorageId`/`targetName`,
  `src/install/install_backend_switch.cpp`). `NcmContentStorage`-API работают
  по открытому хендлу, поэтому проверка свободного места
  (`ncmContentStorageGetFreeSpaceSize`) и запись плейсхолдеров идут по цели без
  правок путей; TempRoot (tik/cert scratch) остаётся на SD. Сообщения об
  ошибках стали target-aware («system memory»/«SD card»). Это заодно закрывает
  задел из 3.5 — вся телеметрия `ncm`/`ncm_stall` теперь честно разделяет `sd`
  vs `nand`.
- Настройка `InstallLocation { SdCard, SystemMemory }` (`app_settings`,
  сериализация `install_location`), UI-селектор «Install location»
  (`src/main_switch.cpp`), проброс через `DownloadManager::setInstallTarget`
  (atomic) → `PackageCoordinator` → `createInstallBackend`. Цель фиксируется
  на старте координатора; трансфер в полёте держит цель, с которой начал.
  Тест: round-trip `install_location` в `tests/test_app_settings.cpp`.

Осталось (device-only, без замера решить нельзя):
- прогнать install на NAND vs SD и сравнить `bps`/`write_us` — поднялся ли
  потолок выше 17 МБ/с;
- (низкое ожидание) быстрый эксперимент 8–16 МиБ чанк;
- если NAND не сдвигает — зафиксировать 16–17 МБ/с как hardware floor SD и
  закрыть 3.3 окончательно (параллелить нечего: крипто ≈4–8% интервала).

### 7.5. Проверить skip delta (3.4) на апдейте с реальными delta в потоке
На замере `deltas=4`, `event=skip` 0. Добавить в телеметрию причину:
delta-id известен, но entry в PFS0 не встречен (нет в NSZ) vs встречен до
cnmt (скип невозможен). Прогнать апдейт, где delta реально лежат в пакете
после cnmt, и убедиться, что skip срабатывает.

### 7.6. SO_RCVBUF: добить остаток
4 отказа «No buffer space available» после handshake. Дёшево: однократный
retry через 1–2 с (пул освобождается после смерти connect-волны) и/или
fallback-запрос 128 КиБ вместо 256. Влияние малое — только для новых пиров
середины прогона.

### 7.7. (опционально) Реюз пировых соединений из magnet-фазы
Magnet-резолв держит живые handshake-соединения, затем торрент строит свои
с нуля: ~10–15 с до первых кусков. Передавать сокеты (или хотя бы список
проверенных endpoint'ов первыми в peer_queue) из резолвера в торрент.
Средняя сложность, экономия фиксированная ~10 с на старт.

# F-C. WebSeed (BEP 19) — HTTP-источники как «пиры» — план реализации

Дата: 2026-07-06. Детализация F-C из `IMPROVEMENT_PLAN.md` (раздел 9).
Мотивация оценена через призму взаимодействия с RuTracker. Ссылки на код —
`файл:строка`.

Суть: загрузчик смешивает ranged-HTTP-источники и обычных пиров в одном
piece-планировщике; данные с HTTP проходят ту же SHA1-верификацию, что и
пиры.

---

## 0. Зачем это для RuTracker

- **Мёртвые и тонкие раздачи — норма для старых игр.** Рой из 0–2 сидов на
  ADSL — типичный кейс; трекер отдаёт пиров, но качать не у кого. WebSeed
  докачивает недостающие куски с HTTP-зеркала при **нуле** живых пиров.
- **РФ-сценарий.** Когда BT-трафик у провайдера зарезан целиком, а
  antizapret-прокси доступен, HTTPS-скачивание — единственный живой канал:
  curl-путь уже умеет проксирование (`antizapret.c:302–345`), announce
  через прокси реализован (RF-план П0.1 ✅). WebSeed превращает это в
  «качает даже при нуле живых пиров» — формулировка самого F-C.
- **Синергия с каталогом.** RuTracker не пишет `url-list` в свои
  .torrent'ы — источник зеркал в нашем случае — каталожная запись
  (`catalog_service.cpp`), которая уже проходит серверный пре-резолв
  info-dict (RF-план П2.1 ✅) и получит подпись (П3.2). Парсер `url-list`
  из .torrent всё равно делаем — для сторонних торрентов.
- Важная оговорка: сам по себе WebSeed не «взаимодействует с RuTracker» —
  он компенсирует слабость его роёв. Здоровье раздач лечит отдача
  (`SEEDING_PLAN.md`); две фичи дополняют друг друга.

## 0.1 Что уже есть в коде

## 1. Дизайн

### 1.1 Модель источника

`webseed_t` (новый `src/core/webseed.c`):

- список URL (из `url-list` метаинфо + зеркала каталога), состояние
  каждого: alive / backoff(t) / dead;
- один рабочий поток на торрент (по образцу announce-worker), в полёте
  1–2 куска;
- задание = **целый кусок** (piece-aligned Range-GET). Гранулярность
  кусок, а не блок: HTTP-запрос не отменяем, дубли с пирами
  минимизируются выбором кусков (см. 1.3), а верификация всё равно
  покусковая.

### 1.2 HTTP-семантика (BEP 19)

- Single-file: `GET url` c `Range: bytes=off-(off+plen-1)`.
- Multi-file (GetRight-style): URL-каталог + `name/path` файла; кусок,
  пересекающий границы файлов, → серия под-запросов по файлам
  (маппинг по `metainfo_t.files`, `metainfo.h`).
- Требования к зеркалу: `Accept-Ranges: bytes`, ответ 206. Ответ 200 на
  ranged-запрос → зеркало dead (иначе скачаем весь файл ради куска).
- Ошибки: 4xx → dead до конца сессии; 5xx/timeout → экспоненциальный
  backoff (1→2→4→… мин), ротация на следующее зеркало.
- TLS/прокси-политика — как у announce (`tracker.c:111–122` +
  `antizapret_apply_route`). Маршрутизация через antizapret — по
  `host_is_target` (`antizapret.c:50`); для зеркал на незаблокированных
  хостах прокси не навязывается.

### 1.3 Интеграция с планировщиком

- Возврат данных: mutex-очередь готовых кусков worker → `torrent_tick()`
  → обычный путь `piece.c` (запись блоков, SHA1, `MSG_HAVE`-broadcast
  после появления отдачи из `SEEDING_PLAN.md`).
- Выбор куска для webseed (в tick, движок однопоточный):
  1. кусок не завершён и **не имеет живых запросов у пиров**
     (`request_counts`, `piece.c:95`);
  2. в strict-order-режиме (stream-install) — первый недоступный кусок
     окна, по которому пиры не укладываются в дедлайн гейта (голова
     стойки — то, что сейчас лечится хеджами, `torrent.c:45`);
  3. в свободном режиме — редчайший кусок (минимальная доступность в
     рое), при мёртвом рое — просто следующий.
- Anti-dup: назначенный webseed'у кусок помечается в `request_counts`;
  пиры на него не наводятся, пока webseed жив. Если кусок пришёл от пира
  раньше — HTTP-результат просто отбрасывается после проверки `have_bf`
  (потерянный трафик ограничен одним куском).
- Гейт: приход HTTP-данных отражается в `RequestGate::onArrived`
  (`request_gate.hpp`) так же, как сетевые куски пиров — бюджет
  reorder-буфера один на всех; отдельно меряется `webseed_bps` (EMA) для
  телеметрии и решения «нужен ли webseed вообще» (при быстром рое —
  спит).

### 1.4 Источники URL

- `metainfo.c`: парсинг `url-list` (строка или список строк, BEP 19) →
  `metainfo_t.webseeds`.
- Каталог: поле `mirrors: [url]` в записи (`catalog_service.cpp`),
  попадает в `torrent_options_t` при старте задачи. Подпись каталога
  (RF-план П3.2) покрывает и зеркала — подмена URL в неподписанном
  каталоге иначе становится вектором (целостность спасает SHA1, но
  логирование IP/контента на чужом зеркале — нет).
- UI: тумблер «HTTP-источники» в настройках (по умолчанию вкл), бейдж
  «web» в карточке задачи + строка скорости.

## 2. Фазы

### Фаза 1 — метаданные и plumbing (S/M)
`url-list` в `metainfo.c` (+ тесты парсера), `mirrors` в каталоге,
прокидка в `torrent_options_t`/`torrent_create_ex`. Кода мало, рисков
нет, разблокирует остальное.

### Фаза 2 — фетчер (M/L)
`webseed.c`: поток, ranged-GET целого куска (single-file), передача в
tick, верификация общим путём, retry/backoff/dead-логика, телеметрия
(`webseed_bps`, `pieces_from_webseed`, счётчики ошибок). Критерий:
торрент с зеркалом и **нулём пиров** докачивается целиком.

### Фаза 3 — смешивание (M)
Правила выбора кусков (§1.3), интеграция с гейтом и strict-order,
multi-file-маппинг, «сон» webseed при быстром рое, отбрасывание дублей.
Критерий: тонкий рой + зеркало быстрее чистого роя, дубль-трафик ≤ 5%.

### Фаза 4 — зеркала и РФ (S/M)
Несколько зеркал с ротацией, antizapret-маршрут для заблокированных
хостов, настройка/бейдж в UI, каталожные записи с зеркалами (процесс
для оператора каталога — вне клиента).

### Фаза 5 — тесты и полировка (S/M)
PC-тест с локальным HTTP-сервером (ranged 206, сценарии: 200-вместо-206,
5xx, обрыв посреди куска, мусорные данные → SHA-fail → dead), fuzz на
парсер `url-list`, golden-обвязка UI-бейджа.

## 3. Риски

| Риск | Вер-ть | Митигация |
|---|---|---|
| Зеркало без корректного Range (200/весь файл) | средняя | детект 206 до чтения тела; dead-mark; тест-кейс |
| Дубль-трафик webseed↔пиры | средняя | гранулярность «кусок», пометка в `request_counts`, webseed спит при здоровом рое |
| Память: +1–2 куска в полёте (4 МиБ каждый) | низкая | лимит 1 кусок в полёте в applet/sysmodule-профиле (бюджет `SYSMODULE_PLAN.md` §2) |
| Злонамеренное зеркало (мусор, tar-бомбы размером) | низкая | SHA1-верификация общим путём; лимит размера ответа = piece_length; dead после N SHA-fail |
| Юридическая сторона хостинга зеркал контента | — | вне скоупа клиента: фича протокольная (BEP 19), зеркала — ответственность оператора каталога; в репо никаких URL по умолчанию |
| curl-поток на торрент + announce-поток: рост числа потоков | низкая | один webseed-поток, только пока задача активна и зеркала alive |

## 4. Критерии приёмки

1. 0 пиров, одно зеркало: торрент качается целиком, скорость ≥ 80%
   прямой HTTP-загрузки того же файла curl'ом на том же железе.
2. Тонкий рой (1 медленный сид ~200 КБ/с) + зеркало: суммарная скорость
   ≥ 2× чистого роя; стрим-установка не встаёт (гейт работает,
   `request_gate.hpp`).
3. Дубль-трафик ≤ 5% от объёма торрента на смешанном прогоне.
4. РФ-профиль: зеркало на заблокированном хосте качается через
   antizapret (интеграционный health-check по образцу ПX.1 из
   `RF_ACCESS_PLAN.md`).
5. Все негативные сценарии фазы 5 проходят без падений и зависаний
   загрузки.

## 5. Не-цели

- BEP 17 (`httpseeds`, устаревший протокол Hoffman) — не реализуем.
- Хостинг/раздача зеркал, наполнение каталога URL'ами.
- HTTP/2, докачка внутри куска после обрыва (обрыв = перезапрос куска).
- Использование webseed для отдачи (это только источник).

## 6. Оценка

Фазы: 1 — S/M, 2 — M/L, 3 — M, 4 — S/M, 5 — S/M. Суммарно **L** —
совпадает с оценкой F-C в `IMPROVEMENT_PLAN.md`. MVP = фазы 1–2
(спасение мёртвых раздач), смешивание и РФ-обвязка — следом.
